## 二、配置与常量管理

### 2.1 路径解析

```python
def get_hermes_home() -> Path:
    return Path(os.getenv("HERMES_HOME", Path.home() / ".hermes"))
```

**渐进式迁移策略**：新安装用统一布局（`cache/images`），已有安装保持旧路径（`image_cache`），零迁移成本。

### 2.2 Profile 多实例

每个 Profile 的 `HERMES_HOME` 指向 `~/.hermes/profiles/<name>`。`get_subprocess_home()` 为 git/ssh/gh 提供独立 HOME。

### 2.3 环境检测

| 检测 | 方法 |
|------|------|
| Termux | `TERMUX_VERSION` 或 `com.termux/files/usr` in PREFIX |
| WSL | `/proc/version` 含 "microsoft" |
| 容器 | `/.dockerenv`、`/run/.containerenv`、cgroup 检测 |

### 2.4 网络偏好

`apply_ipv4_preference()` monkey-patch `socket.getaddrinfo`，解决 IPv6 不可用时的 TCP 超时。

## 四、定时任务系统

### 4.1 架构

```
cron/
  jobs.py        -- Job 存储与 CRUD（~/.hermes/cron/jobs.json）
  scheduler.py   -- 调度执行引擎
```

Gateway 每 60 秒调用 `tick()`，使用文件锁防止重复执行。

### 4.2 调度解析

| 输入 | 解析结果 |
|------|----------|
| `"30m"`, `"2h"` | 一次性，N 分钟后 |
| `"every 30m"` | 循环执行 |
| `"0 9 * * *"` | Cron 表达式 |
| `"2026-02-03T14:00"` | 一次性，指定时间 |

### 4.3 防崩溃重放

执行前先推进 `next_run_at`，崩溃后不会重放。

### 4.4 优雅窗口

- 日常任务：最多 2 小时 catch-up
- 每小时任务：最多 30 分钟

### 4.5 结果投递

支持 18+ 个平台。优先使用 Gateway 的 live adapter，回退到独立 HTTP 发送。

## 五、批处理系统

### 5.1 并行执行

`multiprocessing.Pool` 实现批级并行。并行粒度是**批**（batch），不是单个 prompt。

### 5.2 Checkpoint 机制

增量 checkpoint + 基于内容的 resume（扫描 batch 文件中的实际 prompt 文本）。

### 5.3 工具统计标准化

确保每个条目包含所有可能工具的一致 schema，对 HuggingFace datasets 的 Arrow/Parquet schema 一致性至关重要。

## 十、面试八股文

### 10.1 pyproject.toml vs setup.py

| 维度 | pyproject.toml | setup.py |
|------|---------------|----------|
| 格式 | TOML（标准） | Python 脚本 |
| 声明式 | 是 | 否（可执行代码） |
| 标准化 | PEP 621 | 无正式标准 |
| 构建后端 | 需指定（setuptools/poetry/hatch） | 内置 |

### 10.2 Docker 最佳实践

1. 多阶段构建（减小镜像）
2. 非 root 用户（安全）
3. 最小基础镜像
4. 层缓存优化（COPY requirements 先，COPY 源码后）
5. 健康检查
6. 持久化 volume

### 10.3 定时任务方案对比

| 方案 | 优点 | 缺点 |
|------|------|------|
| APScheduler | 轻量、进程内 | 不支持分布式 |
| Celery Beat | 分布式、可靠 | 重依赖 |
| cron | 系统级、可靠 | 不灵活 |
| Hermes 自建 | 轻量、跨平台 | 功能有限 |

### 10.4 并发编程

| 方式 | 适用场景 |
|------|---------|
| multiprocessing | CPU 密集型 |
| threading | I/O 密集型（受 GIL 限制） |
| asyncio | 高并发 I/O |

### 10.5 12-factor App 配置管理

1. 配置存储在环境变量中
2. 与代码分离
3. 环境间差异最小化

### 10.6 原子写入的三步保障

1. tempfile + fsync（确保数据落盘）
2. os.replace（POSIX 原子替换）
3. BaseException 捕获（清理临时文件）

## 十一、面试官可能问的问题

### Q1: 为什么使用原子写入？

`open("w")` 会立即截断文件。进程崩溃时文件为空。temp-file + os.replace 确保读取者看到完整的旧文件或新文件。

### Q2: Profile 系统如何实现隔离？

每个 Profile 有独立的 `HERMES_HOME`，`get_subprocess_home()` 为子进程提供独立 HOME。配置、技能、记忆、会话完全隔离。

### Q3: 日志系统如何脱敏？

`RedactingFormatter` 在格式化后、写入前，使用正则匹配替换 API keys、tokens 等敏感信息。

### Q4: 定时任务如何防重复执行？

跨平台文件锁（fcntl/msvcrt）。`LOCK_EX | LOCK_NB` 非阻塞排他锁。

### Q5: 定时任务如何防崩溃重放？

执行前先推进 `next_run_at`。即使崩溃，下次 tick 不会重放已完成的任务。

### Q6: 批处理的并行粒度为什么是批不是 prompt？

每个 prompt 需要独立的 AIAgent 实例和完整的 agent loop。并行 prompt 会导致资源竞争。

### Q7: Checkpoint 的基于内容 resume？

不仅靠索引，还扫描 batch 文件中的实际 prompt 文本。即使索引不匹配也能正确恢复。

### Q8: 轨迹压缩的保护策略？

首部的 system/human/gpt/tool 轮次 + 尾部最后 N 轮不压缩。中间区域累积 token 直到满足节省目标。

### Q9: WSL 检测为什么读 /proc/version？

WSL 的内核版本字符串包含 "microsoft"（WSL1）或 "WSL"（WSL2）。

### Q10: IPv4 偏好为什么 monkey-patch getaddrinfo？

某些网络环境 IPv6 不可用但系统优先尝试 IPv6，导致 TCP 超时。Monkey-patch 在 AF_UNSPEC 时优先尝试 IPv4。

### Q11: 日志的会话上下文如何注入？

Record Factory 模式：`logging.setLogRecordFactory()` 注入 `session_tag` 字段，运行在每一条记录上。

### Q12: 噪音抑制如何实现？

在 `setup_logging()` 中将 15 个第三方 logger（openai、httpx、httpcore 等）统一设为 WARNING 级别。

### Q13: 托管模式如何影响权限？

跳过目录权限修改，使用组可读权限 0750/0640。`managed_error()` 打印升级指导。

### Q14: SWE 基准测试的意义？

评估 AI Agent 解决真实软件工程问题的能力。支持 local、docker、modal 三种后端。

### Q15: 如何设计一个可靠的定时任务系统？

1. 文件锁防重复
2. 执行前推进 next_run_at 防重放
3. 优雅窗口处理 catch-up
4. 脚本注入支持数据采集
5. 结果投递到多平台
6. 不活动超时防卡死
