### 2.1 路径解析

`hermes_constants.py` 提供统一的路径解析机制，支持多平台和多Profile隔离。

```python
# 核心实现
def get_hermes_home() -> Path:
    return Path(os.getenv("HERMES_HOME", Path.home() / ".hermes"))

def get_default_hermes_root() -> Path:
    native_home = Path.home() / ".hermes"
    env_home = os.environ.get("HERMES_HOME", "")
    if not env_home:
        return native_home
    env_path = Path(env_home)
    try:
        env_path.resolve().relative_to(native_home.resolve())
        return native_home
    except ValueError:
        pass

    if env_path.parent.name == "profiles":
        return env_path.parent.parent
    
    return env_path
```

```python
def is_termux() -> bool:

def is_wsl() -> bool:
    global _wsl_detected
    if _wsl_detected is not None:
        return _wsl_detected
    try:
        with open("/proc/version", "r") as f:
            _wsl_detected = "microsoft" in f.read().lower()
    except Exception:
        _wsl_detected = False
    return _wsl_detected

def is_container() -> bool:
    global _container_detected
    if _container_detected is not None:
        return _container_detected
    if os.path.exists("/.dockerenv"):
        _container_detected = True
        return True
    if os.path.exists("/run/.containerenv"):
        _container_detected = True
        return True
    try:
        with open("/proc/1/cgroup", "r") as f:
            cgroup = f.read()
            if "docker" in cgroup or "podman" in cgroup or "/lxc/" in cgroup:
                _container_detected = True
                return True
    except OSError:

    _container_detected = False
    return False
```

### 2.5 网络偏好与IPv4优先级
```python
def apply_ipv4_preference(force: bool = False) -> None:
    if not force:
        return

    import socket
    if getattr(socket.getaddrinfo, "_hermes_ipv4_patched", False):
        return

    _original_getaddrinfo = socket.getaddrinfo

    def _ipv4_getaddrinfo(host, port, family=0, type=0, proto=0, flags=0):
        if family == 0:
            try:
                return _original_getaddrinfo(
                    host, port, socket.AF_INET, type, proto, flags
                )
            except socket.gaierror:
                return _original_getaddrinfo(host, port, family, type, proto, flags)
        return _original_getaddrinfo(host, port, family, type, proto, flags)

    _ipv4_getaddrinfo._hermes_ipv4_patched = True
    socket.getaddrinfo = _ipv4_getaddrinfo
```
**设计原理：**
- Monkey-patch系统`socket.getaddrinfo`函数，在请求未指定协议族时优先使用IPv4
- 解决IPv6网络不可用或配置错误导致的TCP连接超时问题
- 安全无侵入：仅在配置`network.force_ipv4: true`时启用
- 幂等设计：多次调用不会重复patch，避免副作用
- 优雅降级：如果IPv4解析失败，自动回退到原解析逻辑，支持纯IPv6环境

### 4.2 调度解析
```python
def parse_schedule(schedule: str) -> Dict[str, Any]:
    schedule = schedule.strip()
    
    # "every X" → 循环间隔
    if schedule_lower.startswith("every "):
        duration_str = schedule[6:].strip()
        minutes = parse_duration(duration_str)
        return {
            "kind": "interval",
            "minutes": minutes,
            "display": f"every {minutes}m"
        }
    
    # Cron表达式（5或6字段）
    parts = schedule.split()
    if len(parts) >= 5 and all(
        re.match(r'^[\d\*\-,/]+$', p) for p in parts[:5]
    ):
        croniter(schedule)  # 验证表达式有效性
        return {
            "kind": "cron",
            "expr": schedule,
            "display": schedule
        }
    
    # ISO时间戳 → 单次执行
    if 'T' in schedule or re.match(r'^\d{4}-\d{2}-\d{2}', schedule):
        dt = datetime.fromisoformat(schedule.replace('Z', '+00:00'))
        if dt.tzinfo is None:
            dt = dt.astimezone()  # 无时区信息视为本地时间
        return {
            "kind": "once",
            "run_at": dt.isoformat(),
            "display": f"once at {dt.strftime('%Y-%m-%d %H:%M')}"
        }
    
    # 持续时间（如30m, 2h, 1d）→ 单次延迟执行
    try:
        minutes = parse_duration(schedule)
        run_at = _hermes_now() + timedelta(minutes=minutes)
        return {
            "kind": "once",
            "run_at": run_at.isoformat(),
            "display": f"once in {original}"
        }
    except ValueError:
        pass
```
**支持的调度格式：**
| 输入示例 | 解析结果 |
|---------|---------|
| `"30m"`, `"2h"`, `"1d"` | 一次性任务，指定时间后执行 |
| `"every 30m"`, `"every 2h"` | 循环任务，按固定间隔执行 |
| `"0 9 * * *"` | Cron表达式，标准5字段格式 |
| `"2026-02-03T14:00"` | 一次性任务，指定时间点执行 |

### 4.3 防崩溃与防重放机制
```python
def advance_next_run(job_id: str) -> bool:
    jobs = load_jobs()
    for job in jobs:
        if job["id"] == job_id:
            kind = job.get("schedule", {}).get("kind")
            if kind not in ("cron", "interval"):
                return False
            now = _hermes_now().isoformat()
            new_next = compute_next_run(job["schedule"], now)
            if new_next and new_next != job.get("next_run_at"):
                job["next_run_at"] = new_next
                save_jobs(jobs)
                return True
            return False
```
**核心设计：**
1. **执行前推进**：周期性任务在执行前先将`next_run_at`推进到下一次时间
2. **原子更新**：任务状态更新使用原子写入，避免部分写入
3. **崩溃安全**：即使进程在执行过程中崩溃，下次重启也不会重复执行
4. **一次性任务例外**：一次性任务不提前推进，允许重启后重试
5. **调度语义**：从"至少一次"转变为"最多一次"，避免重复执行

### 4.4 优雅窗口与错过任务处理

```python
def _compute_grace_seconds(schedule: dict) -> int:
    MIN_GRACE = 120
    MAX_GRACE = 7200  # 2小时

    kind = schedule.get("kind")
    if kind == "interval":
        period_seconds = schedule.get("minutes", 1) * 60
        grace = period_seconds // 2
        return max(MIN_GRACE, min(grace, MAX_GRACE))
    
    if kind == "cron" and HAS_CRONITER:
        try:
            now = _hermes_now()
            cron = croniter(schedule["expr"], now)
            first = cron.get_next(datetime)
            second = cron.get_next(datetime)
            period_seconds = int((second - first).total_seconds())
            grace = period_seconds // 2
            return max(MIN_GRACE, min(grace, MAX_GRACE))
        except Exception:
            pass
    
    return MIN_GRACE
```
**策略：**
- 日常任务：最多2小时catch-up窗口
- 每小时任务：最多30分钟catch-up窗口
- 短间隔任务（<4分钟）：固定120秒窗口
- 超过窗口的错过任务直接跳过，避免重启后任务风暴
- 窗口大小动态计算为周期的一半，在120秒到2小时之间自动调整

## 六、轨迹压缩系统

轨迹压缩系统解决长对话Token超限问题，在保留上下文语义的前提下大幅减少Token使用，支持训练数据压缩和运行时上下文压缩两种场景。

### 6.1 压缩策略
```
┌─────────────────────────────────────────────────────────────┐
│                     COMPRESSION STRATEGY                     │
├─────────────────────────────────────────────────────────────┤
│  保护区域                                                    │
│  • 头部：系统提示、用户初始请求、首条Assistant回复、首次工具调用 │
│  • 尾部：最后N轮对话（默认4轮），保留最近上下文             │
├─────────────────────────────────────────────────────────────┤
│  压缩区域                                                    │
│  • 中间对话轮次：从头开始累积，直到达到需要节省的Token数     │
│  • 压缩为单条摘要消息：由LLM生成，保留关键动作和信息         │
│  • 剩余未压缩轮次保持完整，工具调用历史不变                 │
├─────────────────────────────────────────────────────────────┤
│  压缩保证                                                    │
│  • 压缩后总Token ≤ 目标Token数（默认15250）                  │
│  • 摘要长度目标750 Token，平衡信息密度和上下文大小           │
│  • 工具调用元数据完整保留，不影响后续工具使用               │
└─────────────────────────────────────────────────────────────┘
```

### 6.2 核心实现
```python
def compress_trajectory(
    self, trajectory: List[Dict[str, str]]
) -> Tuple[List[Dict[str, str]], TrajectoryMetrics]:
    # 1. 统计Token分布
    turn_tokens = self.count_turn_tokens(trajectory)
    total_tokens = sum(turn_tokens)
    
    # 2. 无需压缩，直接返回
    if total_tokens <= self.config.target_max_tokens:
        metrics.skipped_under_target = True
        return trajectory, metrics
    
    # 3. 确定可压缩区域
    protected, compress_start, compress_end = self._find_protected_indices(trajectory)
    if compress_start >= compress_end:
        metrics.still_over_limit = True
        return trajectory, metrics
    
    # 4. 计算需要压缩的Token量
    tokens_to_save = total_tokens - self.config.target_max_tokens
    target_tokens_to_compress = tokens_to_save + self.config.summary_target_tokens
    
    # 5. 累积需要压缩的轮次
    accumulated_tokens = 0
    compress_until = compress_start
    for i in range(compress_start, compress_end):
        accumulated_tokens += turn_tokens[i]
        compress_until = i + 1
        if accumulated_tokens >= target_tokens_to_compress:
            break
    
    # 6. 压缩区域不足，压缩全部可压缩区域
    if accumulated_tokens < target_tokens_to_compress and compress_until < compress_end:
        compress_until = compress_end
        accumulated_tokens = sum(turn_tokens[compress_start:compress_end])
    
    # 7. 生成摘要
    content_to_summarize = self._extract_turn_content_for_summary(
        trajectory, compress_start, compress_until
    )
    summary = self._generate_summary(content_to_summarize, metrics)
    
    # 8. 构建压缩后的轨迹
    compressed = []
    # 添加头部轮次，系统提示追加压缩通知
    for i in range(compress_start):
        turn = trajectory[i].copy()
        if turn.get("from") == "system" and self.config.add_summary_notice:
            turn["value"] = turn["value"] + self.config.summary_notice_text
        compressed.append(turn)
    # 添加摘要
    compressed.append({
        "from": "human",
        "value": summary
    })
    # 添加尾部轮次
    for i in range(compress_until, len(trajectory)):
        compressed.append(trajectory[i].copy())
    
    return compressed, metrics
```
**压缩保证：**
- 无信息丢失风险：仅压缩中间轮次，首尾关键上下文完整保留
- Token节省可控：仅压缩到刚好满足目标Token数，最大化保留原始信息
- 摘要质量保证：使用专门的prompt引导模型提取关键动作和信息
- 运行时友好：摘要作为用户消息注入，无需修改Agent逻辑

### 6.3 摘要生成
```python
def _generate_summary(self, content: str, metrics: TrajectoryMetrics) -> str:
    prompt = f"""Summarize the following agent conversation turns concisely. This summary will replace these turns in the conversation history.

Write the summary from a neutral perspective describing what the assistant did and learned. Include:
1. What actions the assistant took (tool calls, searches, file operations)
2. Key information or results obtained
3. Important decisions or findings
4. Relevant data, file names, values, or outputs

Keep the summary factual and informative. Target approximately {self.config.summary_target_tokens} tokens.

---
TURNS TO SUMMARIZE:
{content}
---

Write only the summary, starting with "[CONTEXT SUMMARY]:" prefix."""

    for attempt in range(self.config.max_retries):
        try:
            metrics.summarization_api_calls += 1
            # 使用配置的模型生成摘要，支持OpenRouter等多平台
            response = self.client.chat.completions.create(
                model=self.config.summarization_model,
                messages=[{"role": "user", "content": prompt}],
                temperature=self.config.temperature,
                max_tokens=self.config.summary_target_tokens * 2,
            )
            summary = self._coerce_summary_content(response.choices[0].message.content)
            return self._ensure_summary_prefix(summary)
        except Exception as e:
            metrics.summarization_errors += 1
            if attempt < self.config.max_retries - 1:
                time.sleep(jittered_backoff(attempt + 1, base_delay=self.config.retry_delay, max_delay=30.0))
            else:
                # 降级方案，使用通用摘要
                return "[CONTEXT SUMMARY]: [Summary generation failed - previous turns contained tool calls and responses that have been compressed to save context space.]"
```
**摘要特性：**
- **结构化信息提取**：强制包含动作、结果、决策、关键数据四个维度
- **Token控制**：目标750 Token，生成时限制最大长度为目标的2倍
- **重试机制**：最多3次重试，指数退避，增加成功率
- **降级处理**：API调用失败时返回通用摘要，不中断处理流程

```python
async def _process_directory_async(self, input_dir: Path, output_dir: Path):
    # 信号量控制并发API调用数，避免超限
    semaphore = asyncio.Semaphore(self.config.max_concurrent_requests)
    
    async def process_single(file_path: Path, entry_idx: int, entry: Dict, progress, main_task, status_task):
        nonlocal compressed_count, skipped_count, api_calls, in_flight, timeout_count
        
        async with semaphore:
            async with progress_lock:
                in_flight += 1

                processed_entry, metrics = await asyncio.wait_for(
                    self.process_entry_async(entry),
                    timeout=self.config.per_trajectory_timeout
                )

                    if metrics.was_compressed:
                        compressed_count += 1
                        api_calls += metrics.summarization_api_calls
                    if metrics.skipped_under_target:
                        skipped_count += 1
                    in_flight -= 1
                    progress.advance(main_task)
            except asyncio.TimeoutError:
                timeout_count += 1
                return None
    
    # 创建所有任务并执行
    tasks = [process_single(...) for ... in all_entries]
    await asyncio.gather(*tasks)
```

### 10.6 原子写入的三步保障
```python
def atomic_write(path: Path, content: str):
    # 1. 创建临时文件，fsync确保数据落盘
    temp_path = path.with_suffix(".tmp")
    with open(temp_path, "w") as f:
        f.write(content)
        f.flush()
        os.fsync(f.fileno())
    
    # 2. POSIX原子替换，要么成功要么失败，无中间状态
    os.replace(temp_path, path)
    
    # 3. BaseException捕获，确保临时文件被清理
    try:
        os.unlink(temp_path)
    except:
        pass
```
- **第一步**：写入临时文件并fsync，确保数据真正写入磁盘，不是在页缓存
- **第二步**：os.replace是POSIX标准原子操作，替换过程中不会出现文件不存在或损坏的中间状态
- **第三步**：捕获所有异常（包括KeyboardInterrupt），确保临时文件被清理，避免垃圾文件堆积

## 十一、面试官可能问的问题

### Q1: 为什么使用原子写入？
`open("w")` 模式会立即截断文件，如果进程在写入过程中崩溃，文件会变成空或内容不完整。temp文件+os.replace的原子写入模式保证读取者要么看到完整的旧文件，要么看到完整的新文件，没有中间状态，确保数据一致性。

### Q2: Profile 系统如何实现隔离？
每个Profile有独立的 `HERMES_HOME` 目录，所有配置、技能、记忆、会话数据都存储在该目录下完全隔离。`get_subprocess_home()` 为子进程提供独立的HOME目录，确保git/ssh/gh等工具的配置、密钥完全隔离，不污染系统用户环境。Python进程自身环境不修改，仅在子进程启动时注入环境变量，安全无副作用。

### Q3: 日志系统如何脱敏？
`RedactingFormatter` 在日志格式化后、写入磁盘前，使用多层正则匹配替换敏感信息：
1. 已知API密钥前缀匹配（sk-*, ghp_*, xoxb-*等30+种）
2. 环境变量赋值匹配（API_KEY=*, TOKEN=*等）
3. JSON敏感字段匹配（"apiKey": ".*", "token": ".*"等）
4. Authorization头匹配
5. Telegram bot token、私钥块、数据库连接字符串专项匹配
6. JWT令牌、电话号码、Discord用户ID等隐私信息匹配

短令牌完全替换，长令牌保留前6后4位便于调试，确保敏感信息永远不落盘。

### Q4: 定时任务如何防重复执行？
使用跨平台文件锁（Unix下fcntl，Windows下msvcrt），`LOCK_EX | LOCK_NB` 非阻塞排他锁。如果获取锁失败，说明已有其他进程在执行tick，直接跳过。确保同一时间只有一个进程执行定时任务，避免重复运行。

### Q5: 定时任务如何防崩溃重放？
周期性任务在执行前先将`next_run_at`推进到下一次调度时间，然后再执行任务。即使进程在执行过程中崩溃，下次启动时任务的`next_run_at`已经是未来时间，不会重复执行已经处理过的任务。将调度语义从"至少一次"转变为"最多一次"，避免重复执行带来的副作用。一次性任务不提前推进，允许重启后重试。

### Q6: 批处理的并行粒度为什么是批不是 prompt？
每个Prompt需要独立的AIAgent实例和完整的agent loop，包含多次工具调用和LLM请求。如果并行粒度是单个Prompt，会导致大量并发的LLM API请求和工具进程，容易触发API限流和系统资源竞争。Batch级并行可以更好地控制并发度，每个Batch顺序执行内部Prompt，在吞吐量和资源使用之间取得平衡。

### Q7: Checkpoint 的基于内容 resume？
传统基于索引的resume在数据集修改或排序变化时会错误跳过任务。基于内容的resume通过计算Prompt内容哈希，匹配已完成的任务，即使索引变化或数据集修改也能正确识别已完成的任务。同时支持两种模式，确保在各种场景下都能正确恢复进度。

### Q8: 轨迹压缩的保护策略？
采用首尾保护策略：
1. **头部保护**：系统提示、用户初始请求、首条Assistant回复、首次工具调用，这些是对话的基础上下文，必须完整保留
2. **尾部保护**：最后N轮对话（默认4轮），保留最近的上下文，确保后续回复一致性
3. **中间压缩**：仅压缩中间的对话轮次，从头开始累积直到达到需要节省的Token量，最大化保留最近信息
4. **工具完整性**：所有工具调用的元数据和结果都保留，确保工具状态一致性，不影响后续工具使用

### Q9: WSL 检测为什么读 /proc/version？
WSL1的内核版本字符串包含"Microsoft"标识，WSL2包含"WSL"标识，这是最可靠的跨版本检测方式。其他检测方式如检查`/proc/sys/fs/binfmt_misc/WSLInterop`存在性在某些受限环境中不可用，而读取`/proc/version`几乎在所有环境中都有权限。检测结果进程级缓存，避免重复IO。

### Q10: IPv4 偏好为什么 monkey-patch getaddrinfo？
某些网络环境中IPv6配置错误或不可用，但系统默认优先尝试IPv6，导致TCP连接超时（通常30秒以上）。通过monkey-patch系统`socket.getaddrinfo`函数，在请求未指定协议族时优先返回IPv4地址，可以大幅提升网络请求速度，同时保持对纯IPv6环境的兼容（IPv4解析失败时自动回退）。这种方式对上层应用完全透明，无需修改业务代码。

### Q11: 日志的会话上下文如何注入？
使用`logging.setLogRecordFactory()`替换全局日志记录工厂，在每个LogRecord创建时自动注入`session_tag`字段，值为当前线程的`session_id`（如果存在）。这种方式比Filter更可靠，因为Filter是Handler级别的，而记录工厂是全局的，所有日志（包括第三方库日志）都会包含session_tag，实现全链路会话跟踪。

### Q12: 噪音抑制如何实现？
在`setup_logging()`中将15+个常用第三方库的日志级别统一设为WARNING，包括openai、httpx、httpcore、asyncio、grpc等。这些库在INFO/DEBUG级别会产生大量无用日志，抑制后可以保持日志简洁，突出业务相关信息。verbose模式下仍然可以临时启用DEBUG级日志。

### Q13: 托管模式如何影响权限？
托管模式（如NixOS部署）中，状态目录使用setgid权限（2770），确保新创建的文件继承目录的组所有权。自定义`_ManagedRotatingFileHandler`在文件创建和日志轮转时自动将权限设为0o660，确保网关进程和交互式CLI用户（同组）都能读写日志文件，避免权限错误。

### Q14: SWE 基准测试的意义？
SWE-bench是行业标准的软件工程Agent基准测试，包含真实的GitHub issues和代码库，评估AI Agent解决实际软件工程问题的能力。Hermes支持三种后端运行测试：
1. **本地后端**：直接在本地机器运行，速度快，适合开发调试
2. **Docker后端**：每个任务在独立Docker容器中运行，环境隔离，安全
3. **Modal后端**：云端分布式运行，支持大规模并行测试，速度快

测试结果自动生成分数和详细报告，便于迭代改进Agent能力。

### Q15: 如何设计一个可靠的定时任务系统？
1. **文件锁防重复**：跨平台文件锁确保同一时间只有一个调度实例运行
2. **执行前推进调度时间**：防崩溃重放，避免重复执行
3. **优雅窗口处理错过的任务**：避免重启后的任务风暴
4. **内置脚本注入支持**：任务运行前执行自定义脚本，动态注入上下文
5. **多平台结果投递**：支持18+个消息平台，任务结果自动投递
6. **不活动超时机制**：检测无活动的挂起任务，自动终止，避免资源泄漏
7. **原子状态更新**：任务状态持久化使用原子写入，避免数据损坏
8. **幂等设计**：任务执行应该是幂等的，即使重复执行也不会产生副作用
