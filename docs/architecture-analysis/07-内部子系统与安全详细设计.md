## 1. 凭证池 (CredentialPool)

**文件**: `agent/credential_pool.py` (~1419行)

凭证池是 Hermes Agent 实现**高可用API调用**的核心基础设施。当一个凭证耗尽（429限流/402额度用尽）时，系统自动切换到下一个可用凭证，实现无缝failover。

### 1.1 PooledCredential 数据结构

```python
@dataclass
class PooledCredential:
    provider: str                          # 所属提供者 (anthropic/openrouter/nous等)
    id: str                                # 唯一标识 (uuid hex[:6])
    label: str                             # 人类可读标签 (email/JWT claims)
    auth_type: str                         # "oauth" 或 "api_key"
    priority: int                          # 优先级排序 (0=最高)
    source: str                            # 来源: "manual"/"claude_code"/"hermes_pkce"/"env:ANTHROPIC_API_KEY"
    access_token: str                      # 当前访问令牌
    refresh_token: Optional[str]           # OAuth刷新令牌 (一次性使用)
    last_status: Optional[str]             # "ok" / "exhausted"
    last_status_at: Optional[float]        # 状态变更时间戳
    last_error_code: Optional[int]         # HTTP状态码 (429/402等)
    last_error_reason: Optional[str]       # 错误原因分类
    last_error_message: Optional[str]      # 原始错误消息
    last_error_reset_at: Optional[float]   # 提供者返回的重置时间戳
    base_url: Optional[str]                # API端点URL
    expires_at_ms: Optional[int]           # OAuth令牌过期时间 (毫秒)
    request_count: int = 0                 # 已处理请求数 (用于least_used策略)
    extra: Dict[str, Any]                  # 扩展字段 (token_type/scope/client_id等)
```

**设计亮点**:
- `extra` 字典通过 `__getattr__` 动态代理，访问 `entry.client_id` 等价于 `entry.extra["client_id"]`，保持向后兼容
- `_EXTRA_KEYS` 集合声明了12个仅用于JSON序列化、不参与逻辑判断的字段
- `runtime_api_key` 属性对 Nous 提供者特殊处理：优先使用 `agent_key` 而非 `access_token`
- `from_dict()` 工厂方法自动为缺失字段设置合理默认值

### 1.2 四种选择策略

通过 `config.yaml` 的 `credential_pool_strategies` 配置：

| 策略 | 行为 | 适用场景 |
|------|------|----------|
| `fill_first` (默认) | 始终选择优先级最高的可用凭证 | 单主力+备用凭证 |
| `round_robin` | 轮转：每次选择后将当前凭证移到队尾 | 多同级凭证均匀分摊 |
| `random` | 随机选择可用凭证 | 测试/快速均衡 |
| `least_used` | 选择 `request_count` 最小的凭证 | 精确负载均衡 |

```python
def _select_unlocked(self) -> Optional[PooledCredential]:
    available = self._available_entries(clear_expired=True, refresh=True)
    if self._strategy == STRATEGY_RANDOM:
        entry = random.choice(available)
    elif self._strategy == STRATEGY_LEAST_USED and len(available) > 1:
        entry = min(available, key=lambda e: e.request_count)
    elif self._strategy == STRATEGY_ROUND_ROBIN and len(available) > 1:
        # 旋转：将第一个移到最后，重新编号优先级
        rotated = [c for c in self._entries if c.id != entry.id]
        rotated.append(replace(entry, priority=len(self._entries) - 1))
        self._entries = [replace(c, priority=idx) for idx, c in enumerate(rotated)]
    else:  # fill_first
        entry = available[0]
```

**round_robin 的关键实现细节**: 通过修改 `priority` 字段并持久化，实现跨进程的状态传递。每次选择后重新编号所有条目的优先级，保证下次重启后仍然从正确位置继续轮转。

**刷新失败的重试策略**:
1. Anthropic: 先刷新 → 失败后从 credentials.json 同步 → 若令牌变化则重试 → 若文件令牌未过期直接使用
2. Codex: 先从 ~/.codex/auth.json 同步 → 再刷新 → 失败后再次同步重试
3. 所有路径均在失败后标记 `exhausted`，等待冷却期恢复

### 1.6 并发租约机制

```python
DEFAULT_MAX_CONCURRENT_PER_CREDENTIAL = 1

def acquire_lease(self, credential_id=None) -> Optional[str]:
    """软租约：不阻塞，即使超过上限也返回最不繁忙的凭证"""
    below_cap = [e for e in available
                 if self._active_leases.get(e.id, 0) < self._max_concurrent]
    candidates = below_cap if below_cap else available
    chosen = min(candidates, key=lambda e: (
        self._active_leases.get(e.id, 0),  # 最少活跃租约
        e.priority                           # 优先级作为稳定tie-breaker
    ))

def release_lease(self, credential_id: str) -> None:
    """释放租约，计数器减1"""
```

**设计哲学**: 软上限（soft cap）而非硬阻塞。即使所有凭证都达到并发上限，仍然选择最不繁忙的返回，保证系统不会死锁。

### 1.7 跨进程同步与持久化

所有状态变更通过 `_persist()` → `write_credential_pool()` 写入 `~/.hermes/auth.json`。

**`_seed_from_singletons()` 初始化流程**:
1. 从 auth.json 加载已有池条目
2. 从单例凭证（环境变量、Claude Code文件、Nous device_code）补充
3. 修剪过期的seed条目（来源不再存在的）
4. Anthropic特定：规范化优先级（手动 > hermes_pkce > claude_code > env:API_KEY）

### 1.8 父子Agent共享

凭证池通过文件系统 (`auth.json`) 实现跨进程共享。当父Agent刷新令牌后：
1. `_sync_device_code_entry_to_auth_store()` 将新令牌写回 auth.json 的 providers 节点
2. 子Agent的 `load_pool()` → `_seed_from_singletons()` 读到新鲜状态
3. 防止子Agent用过期的单例令牌覆盖父Agent刚刷新的池条目

### 1.9 自定义端点池

支持 `custom:<name>` 格式的池键，自动从两个来源初始化：
- `config.yaml` 的 `custom_providers` 列表中的 `api_key`
- `model.api_key` (当 `model.provider == "custom"` 且 base_url 匹配时)

### 2.1 FailoverReason 枚举 (15种)

```python
class FailoverReason(enum.Enum):
    # 认证/授权
    auth = "auth"                           # 401/403 — 临时认证失败
    auth_permanent = "auth_permanent"       # 刷新后仍失败 — 放弃

    # 计费/配额
    billing = "billing"                     # 402 额度耗尽 — 立即轮转
    rate_limit = "rate_limit"               # 429 限流 — 退避后轮转

    # 服务端
    overloaded = "overloaded"               # 503/529 — 服务过载
    server_error = "server_error"           # 500/502 — 内部错误

    # 传输层
    timeout = "timeout"                     # 连接/读取超时

    # 上下文/负载
    context_overflow = "context_overflow"   # 上下文过大 — 压缩而非failover
    payload_too_large = "payload_too_large" # 413 — 压缩负载

    # 模型
    model_not_found = "model_not_found"     # 404/无效模型

    # 请求格式
    format_error = "format_error"           # 400 — 格式错误

    # 提供者特定
    thinking_signature = "thinking_signature"  # Anthropic思考块签名无效
    long_context_tier = "long_context_tier"    # Anthropic长上下文tier门控

    # 兜底
    unknown = "unknown"                     # 无法分类 — 退避重试
```

### 2.2 ClassifiedError 数据结构

### 2.3 7层分类流水线

```
优先级从高到低:

Layer 1: 提供者特定模式 (最高优先)
  ├── Anthropic thinking block签名 (400 + "signature" + "thinking")
  └── Anthropic long-context tier门控 (429 + "extra usage" + "long context")

Layer 2: HTTP状态码 + 消息感知细化
  ├── 401 → auth (retryable=false, rotate+fallback)
  ├── 403 → auth (特殊: "key limit exceeded" → billing)
  ├── 402 → 区分临时配额 vs 确认耗尽 (见下文)
  ├── 404 → model_not_found
  ├── 413 → payload_too_large (compress)
  ├── 429 → rate_limit (rotate+fallback)
  ├── 400 → context_overflow / format_error / 402伪装 / 大会话暗示溢出
  ├── 500/502 → server_error
  └── 503/529 → overloaded

Layer 3: 错误码分类 (response body中的error.code)
  ├── resource_exhausted / throttled → rate_limit
  ├── insufficient_quota / billing_not_active → billing
  ├── model_not_found → model_not_found
  └── context_length_exceeded → context_overflow

Layer 4: 消息模式匹配 (无状态码时)
  ├── 负载过大模式
  ├── 用量限制 + 暂态信号 → rate_limit
  ├── 用量限制无暂态信号 → billing
  ├── 账单模式
  ├── 限流模式
  ├── 上下文溢出模式 (20+模式，含中文/中文错误消息)
  ├── 认证模式
  └── 模型未找到模式

Layer 5: 服务端断连 + 大会话 → 上下文溢出
  (比通用传输层错误更优先：大session的RemoteProtocolError更可能是上下文溢出)
  判定条件: approx_tokens > context_length * 0.6 OR > 120K OR messages > 200

Layer 6: 传输/超时启发式
  (ReadTimeout, ConnectError, APIConnectionError等Python异常类型)

Layer 7: 兜底 — unknown (retryable=true)
```

### 2.4 402歧义消除

这是最精巧的设计之一。402既可能是账单耗尽，也可能是临时配额：

```python
def _classify_402(error_msg, result_fn):
    has_usage_limit = any(p in error_msg for p in _USAGE_LIMIT_PATTERNS)
    has_transient = any(p in error_msg for p in _USAGE_LIMIT_TRANSIENT_SIGNALS)

    if has_usage_limit and has_transient:
        # "Usage limit, try again in 5 minutes" → rate_limit (可恢复)
        return result_fn(FailoverReason.rate_limit, retryable=True, ...)
    # 确认账单耗尽
    return result_fn(FailoverReason.billing, retryable=False, ...)
```

## 5. 上下文压缩算法

**文件**: `agent/context_compressor.py` (~1092行)

### 5.1 算法总览

```
Phase 1: 工具输出修剪 (廉价预处理，无LLM调用)
  ├── Pass 1: 去重 — 相同工具输出只保留最新完整副本
  ├── Pass 2: 修剪旧工具结果 → 信息性一行摘要
  └── Pass 3: 截断大工具调用参数 (>500字符 → 200字符)

Phase 2: 确定边界
  ├── 保护头部: 系统提示 + 前N条消息 (默认3)
  ├── 保护尾部: 按token预算 (~20K tokens) 从尾部回溯
  └── 对齐边界: 不切断 tool_call/tool_result 对

Phase 3: LLM摘要生成
  ├── 结构化模板 (Goal/Progress/Decisions/Resolved/Pending/Files/Remaining Work)
  ├── 迭代更新 (存在前次摘要时增量合并)
  └── 聚焦主题 (/compress <topic> 时60-70%预算给主题)

Phase 4: 组装压缩消息列表
  ├── 保持角色交替 (user/assistant严格交替)
  ├── 合并到尾部 (当无法保持交替时)
  └── 清理孤立的tool_call/tool_result对
```

## 10. 面试技术亮点

### 10.1 凭证池的跨进程令牌同步

**问题**: OAuth refresh_token是一次性的。当Claude Code CLI和Hermes同时运行时，谁先刷新谁就消费了token，另一个会得到 "refresh_token_reused" 错误。

**解决方案**: 三方同步机制 — 池条目 ↔ credentials.json ↔ auth.json providers:
1. 刷新前先从文件同步最新token
2. 刷新后写回所有相关文件
3. 失败时检测文件是否已被其他进程更新，若是则重试

### 10.2 402的临时配额 vs 账单耗尽辨别

**问题**: 某些提供者对临时配额也返回402，直接按"账单耗尽"处理会导致不必要的凭证轮转。

**解决方案**: 双信号检测 — `USAGE_LIMIT_PATTERNS` (配额关键词) + `USAGE_LIMIT_TRANSIENT_SIGNALS` (暂态信号如 "try again"/"resets at"/"wait")。两个信号同时出现才判定为临时配额。

### 10.3 Thinking Block签名管理

**问题**: Anthropic对thinking block签名。任何上游修改（上下文压缩、会话截断、孤立块剥离、消息合并）都会使签名失效，导致400错误。

**解决方案**: 三层策略:
1. 第三方端点: 剥离所有thinking块（签名是Anthropic专有的）
2. Anthropic直接端点: 只保留最新assistant消息的有签名thinking
3. 无签名thinking: 降级为纯文本（避免信息丢失）
4. 剥离所有thinking块的cache_control（缓存标记干扰签名验证）

### 10.4 上下文压缩的反抖动机制

**问题**: 当对话几乎全是不可压缩内容时，每次压缩只省1-2%，触发无限循环。

**解决方案**: 连续两次压缩节省不到10%时暂停压缩，建议用户 `/new` 开始新会话或 `/compress <topic>` 做聚焦压缩。

### 10.5 辅助客户端的7级回退链

**问题**: 辅助任务（压缩/搜索/视觉）需要独立于主模型的轻量后端，但用户的配置可能只有一种提供者。

**解决方案**: 7级自动探测链，覆盖从OpenRouter到本地端点的所有可能路径，402信用耗尽时自动跳到下一级。

### 10.6 模型元数据的10级上下文长度解析

**问题**: 不同提供者/端点对同一模型的上下文长度限制不同（如Claude Opus 4.6在Anthropic是1M，在Copilot是128K）。

**解决方案**: 10级回退链，包含provider-aware查找、本地服务器探测、持久化缓存、硬编码默认值。每次探测成功后写入 `context_length_cache.yaml`，避免重复网络调用。

### 10.7 注入检测的阻断策略

**问题**: 用户项目中的 `.cursorrules` 或 `AGENTS.md` 可能包含恶意提示注入。

**解决方案**: 在文件内容注入system prompt之前，用10个正则模式 + 不可见字符检测扫描。发现威胁时完全阻断（不加载），而非尝试净化（可能遗漏），返回明确的BLOCKED标记。

### 10.8 Codex Responses → chat.completions 适配

**问题**: Codex OAuth只能访问Responses API，但所有辅助消费者都用chat.completions接口。

**解决方案**: `CodexAuxiliaryClient` 包装层将 `client.chat.completions.create()` 调用翻译为 Responses API，包括:
- system → instructions 分离
- content格式转换 (text → input_text, image_url → input_image)
- 流式事件收集 + 空输出回填
- function_call → tool_calls 转换
- 异步版本通过 `asyncio.to_thread()` 桥接
