## 一、模块概述

Hermes Agent 的工具系统是一个高度模块化的插件式架构，核心设计哲学：**自注册、自发现**。

### 四层架构
```
┌─────────────────────────────────────────────────────┐
│  model_tools.py  —— 编排层                           │
│  负责触发发现、提供公共API、桥接同步/异步              │
├─────────────────────────────────────────────────────┤
│  toolsets.py  —— 工具集层                            │
│  定义工具分组、递归解析、组合继承                      │
├─────────────────────────────────────────────────────┤
│  tools/registry.py  —— 注册中心                      │
│  单例注册表、schema管理、分发执行                      │
├─────────────────────────────────────────────────────┤
│  tools/*.py  —— 工具实现层                           │
│  每个文件自包含schema+handler+注册调用                 │
└─────────────────────────────────────────────────────┘
```

### 导入链设计（循环导入安全）
```
tools/registry.py  (不导入 model_tools 或工具文件)
       ^
tools/*.py  (在模块级别导入 tools.registry)
       ^
model_tools.py  (导入 tools.registry + 所有工具模块)
       ^
run_agent.py, cli.py, batch_runner.py, etc.
```

### 核心设计优势
1. **开闭原则**：新增工具无需修改现有核心代码
2. **内聚性**：工具的schema、handler、依赖检查在同一文件
3. **自动发现**：AST扫描识别registry.register调用，无需手动维护工具列表
4. **扩展性**：支持MCP外部工具、插件工具动态加载
5. **隔离性**：工具之间不直接依赖，通过注册表通信

## 二、ToolRegistry 详解

### 2.1 单例模式实现

模块级单例：
```python
# registry.py
registry = ToolRegistry()
```

Python 的模块系统天然保证每个模块只被导入一次（`sys.modules` 缓存），因此是天然的单例。线程安全由 import lock 保证。

### 2.2 AST 扫描自动发现机制

```python
def discover_builtin_tools(tools_dir=None):
    tools_path = Path(tools_dir) if tools_dir else Path(__file__).resolve().parent
    module_names = [
        f"tools.{path.stem}"
        for path in sorted(tools_path.glob("*.py"))
        if path.name not in {"__init__.py", "registry.py", "mcp_tool.py"}
        and _module_registers_tools(path)
    ]
```

`_module_registers_tools()` 用 AST 分析精确匹配 `registry.register(...)` 调用模式：

```python
def _is_registry_register_call(node: ast.AST) -> bool:
    if not isinstance(node, ast.Expr) or not isinstance(node.value, ast.Call):
        return False
    func = node.value.func
    return (
        isinstance(func, ast.Attribute)
        and func.attr == "register"
        and isinstance(func.value, ast.Name)
        and func.value.id == "registry"
    )
```

**为什么用 AST 而不是正则**：
1. 精确性：AST 理解 Python 语法，不会被注释、字符串干扰
2. 结构化：只检查顶层语句，避免函数内部调用的误匹配
3. 安全性：只解析不执行，不会触发模块副作用

### 2.3 register() 方法

```python
def register(
    self,
    name: str,           # 工具唯一名称
    toolset: str,        # 所属工具集
    schema: dict,        # OpenAI function-calling 格式的 schema
    handler: Callable,   # 工具执行函数
    check_fn: Callable = None,       # 可用性检查函数
    requires_env: list = None,       # 依赖的环境变量
    is_async: bool = False,          # 是否为异步处理器
    description: str = "",
    emoji: str = "",
    max_result_size_chars: int | float | None = None,
):
```

**防覆盖机制**：当同名工具从不同 toolset 注册时，系统拒绝覆盖内置工具，但允许 MCP 工具之间的覆盖。


### 2.5 Schema 生成和验证

`get_definitions()` 将注册的工具转换为 OpenAI function-calling 格式。使用 `check_results` 缓存机制：同一个 `check_fn` 只调用一次。

```python
def get_definitions(self, tool_names: Set[str], quiet: bool = False) -> List[dict]:
    result = []
    check_results: Dict[Callable, bool] = {}  # check_fn结果缓存
    entries_by_name = {entry.name: entry for entry in self._snapshot_entries()}
    
    for name in sorted(tool_names):
        entry = entries_by_name.get(name)
        if not entry:
            continue
        if entry.check_fn:
            if entry.check_fn not in check_results:
                try:
                    check_results[entry.check_fn] = bool(entry.check_fn())
                except Exception:
                    check_results[entry.check_fn] = False
            if not check_results[entry.check_fn]:
                continue
        schema_with_name = {**entry.schema, "name": entry.name}
        result.append({"type": "function", "function": schema_with_name})
    return result
```

### 2.6 分发执行

```python
def dispatch(self, name: str, args: dict, **kwargs) -> str:
    entry = self.get_entry(name)
    if not entry:
        return json.dumps({"error": f"Unknown tool: {name}"})
    try:
        if entry.is_async:
            from model_tools import _run_async
            return _run_async(entry.handler(args, **kwargs))
        return entry.handler(args, **kwargs)
    except Exception as e:
        return json.dumps({"error": f"Tool execution failed: {e}"})
```

### 2.7 线程安全设计

使用 `threading.RLock()`（可重入锁）保护所有修改操作。读操作使用**快照隔离**模式。

```python
def _snapshot_state(self) -> tuple[List[ToolEntry], Dict[str, Callable]]:
    """返回注册表条目的一致快照，避免读写竞争"""
    with self._lock:
        return list(self._tools.values()), dict(self._toolset_checks)
```

### 3.2 递归解析机制

```python
def resolve_toolset(name: str, visited: Set[str] = None) -> List[str]:
    if visited is None:
        visited = set()
    if name in {"all", "*"}:
        all_tools = set()
        for toolset_name in get_toolset_names():
            resolved = resolve_toolset(toolset_name, visited.copy())
            all_tools.update(resolved)
        return sorted(all_tools)
    if name in visited:
        return []  # 循环检测
    visited.add(name)
    toolset = get_toolset(name)
    tools = set(toolset.get("tools", []))
    for included_name in toolset.get("includes", []):
        included_tools = resolve_toolset(included_name, visited)
        tools.update(included_tools)
    return sorted(tools)
```

关键设计：循环检测、菱形依赖安全、"all" 别名。

### 3.3 工具集动态扩展

```python
def get_all_toolsets() -> Dict[str, Dict[str, Any]]:
```

## 四、编排层 (model_tools.py) 核心实现

### 4.1 异步桥接设计

解决同步上下文调用异步工具的问题，支持三种执行路径：

```python
def _run_async(coro):
    """Run an async coroutine from a sync context."""
    try:
        loop = asyncio.get_running_loop()
    except RuntimeError:
        loop = None

    if loop and loop.is_running():
        # 1. 已有运行中循环 → 新线程异步执行
        import concurrent.futures
        with concurrent.futures.ThreadPoolExecutor(max_workers=1) as pool:
            future = pool.submit(asyncio.run, coro)
            return future.result(timeout=300)

    if threading.current_thread() is not threading.main_thread():
        # 2. 工作线程 → 线程本地持久loop
        worker_loop = _get_worker_loop()
        return worker_loop.run_until_complete(coro)

    # 3. 主线程无循环 → 全局持久tool_loop
    tool_loop = _get_tool_loop()
    return tool_loop.run_until_complete(coro)
```

**持久Loop设计优势**：避免重复创建关闭loop导致的"Event loop is closed"错误，保持HTTP客户端连接复用。

### 4.3 工具参数类型强制

LLM经常返回字符串形式的数字/布尔值，coerce_tool_args自动转换为schema声明的类型：

```python
def coerce_tool_args(tool_name: str, args: Dict[str, Any]) -> Dict[str, Any]:
    """Coerce tool call arguments to match their JSON Schema types."""
    if not args or not isinstance(args, dict):
        return args

    schema = registry.get_schema(tool_name)
    if not schema:
        return args

    properties = (schema.get("parameters") or {}).get("properties")
    if not properties:
        return args

    for key, value in args.items():
        if not isinstance(value, str):
            continue
        prop_schema = properties.get(key)
        if not prop_schema:
            continue
        expected = prop_schema.get("type")
        if not expected:
            continue
        coerced = _coerce_value(value, expected)
        if coerced is not value:
            args[key] = coerced

    return args
```

支持类型：
- integer: "42" → 42
- number: "3.14" → 3.14
- boolean: "true" → True, "false" → False
- 联合类型：尝试所有类型直到成功

### 4.6 并行工具执行

run_agent.py 支持将互不依赖的工具调用并发执行，大幅减少多工具调用轮次的延迟。

**三层安全分类**：

```python
# run_agent.py L216 — 绝对禁止并行的工具
_NEVER_PARALLEL_TOOLS = frozenset({"clarify"})
# clarify 需要同步用户交互，并行执行会产生冲突

# run_agent.py L219-231 — 始终可并行的只读工具
_PARALLEL_SAFE_TOOLS = frozenset({
    "ha_get_state", "ha_list_entities", "ha_list_services",
    "read_file", "search_files", "session_search",
    "skill_view", "skills_list", "vision_analyze",
    "web_extract", "web_search",
})
# 这些工具无共享可变状态，读写不冲突

# run_agent.py L234 — 路径范围内有条件的并行
_PATH_SCOPED_TOOLS = frozenset({"read_file", "write_file", "patch"})
# 仅当操作的文件路径不重叠时才能并发
```

**并行决策逻辑 `_should_parallelize_tool_batch()`**（run_agent.py L267-308）：

1. 单工具调用直接跳过（`len(tool_calls) <= 1`）
2. 检查是否包含 `_NEVER_PARALLEL_TOOLS` → 有则回退串行
3. 解析每个工具的参数（JSON parse 失败则回退串行）
4. 对 `_PATH_SCOPED_TOOLS`，提取目标文件路径，通过 `_paths_overlap()` 检查路径是否冲突
5. 非路径工具必须在 `_PARALLEL_SAFE_TOOLS` 中，否则回退串行

```python
def _paths_overlap(a: Path, b: Path) -> bool:
    """两个路径重叠当且仅当它们相同或一个是另一个的父目录。"""
    a = a.resolve()
    b = b.resolve()
    return a == b or a in b.parents or b in a.parents
```

**执行引擎**（`_execute_tool_calls_concurrent`）：

```python
_MAX_TOOL_WORKERS = 8  # ThreadPoolExecutor 最大并发数
max_workers = min(num_tools, _MAX_TOOL_WORKERS)
```

- 使用 `ThreadPoolExecutor` 并发执行工具调用
- 每个 worker 线程设置 `activity_callback`，使得长时间运行的 terminal 命令能发送心跳
- 主线程每 30 秒轮询一次 `concurrent.futures.wait()`，期间调用 `_touch_activity()` 防止网关超时
- 结果按原始 tool_call 顺序收集，确保 API 看到正确的 tool message 序列
- `_invoke_tool()` 本身有 try/except 保护，异常不会导致线程崩溃

## 五、工具实现层规范

### 5.1 最简工具示例 (clarify_tool.py)

```python
#!/usr/bin/env python3
"""Clarify Tool Module - Interactive Clarifying Questions"""

import json
from typing import List, Optional, Callable
from tools.registry import registry, tool_error

MAX_CHOICES = 4

def clarify_tool(
    question: str,
    choices: Optional[List[str]] = None,
    callback: Optional[Callable] = None,
) -> str:
    """Ask the user a question, optionally with multiple-choice options."""
    if not question or not question.strip():
        return tool_error("Question text is required.")
    
    question = question.strip()
    
    # Validate choices
    if choices is not None:
        choices = [str(c).strip() for c in choices if str(c).strip()]
        if len(choices) > MAX_CHOICES:
            choices = choices[:MAX_CHOICES]
        if not choices:
            choices = None
    
    if callback is None:
        return json.dumps(
            {"error": "Clarify tool is not available in this execution context."},
            ensure_ascii=False,
        )
    
    try:
        user_response = callback(question, choices)
    except Exception as exc:
        return json.dumps(
            {"error": f"Failed to get user input: {exc}"},
            ensure_ascii=False,
        )
    
    return json.dumps({
        "question": question,
        "choices_offered": choices,
        "user_response": str(user_response).strip(),
    }, ensure_ascii=False)

def check_clarify_requirements() -> bool:
    """Clarify tool has no external requirements -- always available."""
    return True

# OpenAI Function-Calling Schema
CLARIFY_SCHEMA = {
    "name": "clarify",
    "description": "Ask the user a question when you need clarification...",
    "parameters": {
        "type": "object",
        "properties": {
            "question": {
                "type": "string",
                "description": "The question to present to the user.",
            },
            "choices": {
                "type": "array",
                "items": {"type": "string"},
                "maxItems": MAX_CHOICES,
                "description": "Up to 4 answer choices...",
            },
        },
        "required": ["question"],
    },
}

# 注册工具
registry.register(
    name="clarify",
    toolset="clarify",
    schema=CLARIFY_SCHEMA,
    handler=lambda args, **kw: clarify_tool(
        question=args.get("question", ""),
        choices=args.get("choices"),
        callback=kw.get("callback")),
    check_fn=check_clarify_requirements,
    emoji="❓",
)
```

### 5.2 工具实现规范

每个工具文件必须包含：
1. **Schema定义**：遵循OpenAI function-calling格式
2. **Handler函数**：工具实际执行逻辑，返回JSON字符串
3. **Check函数**：可用性检查，返回bool
4. **注册调用**：调用registry.register完成自注册

```python
return tool_result(success=True, data=payload)

# 错误返回
return tool_error("Something went wrong")
# 或带额外信息
return tool_error("Permission denied", code=403)
```

## 八、多Provider Schema转换

### 8.1 核心设计：OpenAI 格式作为规范中间表示

Hermes 的工具 schema 以 OpenAI function-calling 格式为唯一的注册格式（即 `{name, description, parameters}` 结构），所有 Provider 适配器从这一源头转换到各自的原生格式。

```python
# tools/registry.py — get_definitions() 输出 OpenAI 格式作为规范格式
def get_definitions(self, tool_names: Set[str], quiet: bool = False) -> List[dict]:
    schema_with_name = {**entry.schema, "name": entry.name}
    result.append({"type": "function", "function": schema_with_name})
    return result
```

**为什么一个源头多个目标是架构上合理的**：

1. **单一事实来源**：schema 只在一处定义（工具文件中的 `XXX_SCHEMA`），避免多份 schema 不同步
2. **Provider 无关的工具实现**：工具 handler 不需要知道 LLM 用的是 OpenAI 还是 Anthropic 格式
3. **新增 Provider 成本低**：只需实现一个转换器（~30 行代码），不必修改任何工具文件
4. **分离关注点**：schema 的结构语义（OpenAI）与 Provider 的 wire format 各司其职

### 8.2 Anthropic Schema 转换

`convert_tools_to_anthropic()` 在 `agent/anthropic_adapter.py` L806 实现：

```python
def convert_tools_to_anthropic(tools: List[Dict]) -> List[Dict]:
    """Convert OpenAI tool definitions to Anthropic format."""
    result = []
    for t in tools:
        fn = t.get("function", {})
        result.append({
            "name": fn.get("name", ""),
            "description": fn.get("description", ""),
            "input_schema": fn.get("parameters", {"type": "object", "properties": {}}),
        })
    return result
```

转换映射：`{type: "function", function: {name, description, parameters}}` → `{name, description, input_schema}`

在 `agent/anthropic_adapter.py` L1266 使用：
```python
anthropic_tools = convert_tools_to_anthropic(tools) if tools else []
```

### 8.3 MCP Schema 转换

MCP（Model Context Protocol）工具的定义格式与 Hermes 注册表格式不同，需要专门的转换层。所有转换逻辑集中在 `tools/mcp_tool.py` 中。

**`_convert_mcp_schema()`**（tools/mcp_tool.py L1652）将 MCP Tool 对象转换为 registry 格式：

```python
def _convert_mcp_schema(server_name: str, mcp_tool) -> dict:
    safe_tool_name = sanitize_mcp_name_component(mcp_tool.name)
    safe_server_name = sanitize_mcp_name_component(server_name)
    prefixed_name = f"mcp_{safe_server_name}_{safe_tool_name}"
    return {
        "name": prefixed_name,
        "description": mcp_tool.description or f"MCP tool {mcp_tool.name} from {server_name}",
        "parameters": _normalize_mcp_input_schema(mcp_tool.inputSchema),
    }
```

**`_normalize_mcp_input_schema()`**（tools/mcp_tool.py L1630）处理 input schema 的边界情况：

```python
def _normalize_mcp_input_schema(schema: dict | None) -> dict:
    """Normalize MCP input schemas for LLM tool-calling compatibility."""
    if not schema:
        return {"type": "object", "properties": {}}
    if schema.get("type") == "object" and "properties" not in schema:
        return {**schema, "properties": {}}
    return schema
```

关键处理：
- 空 schema → 补全 `{type: "object", properties: {}}`
- 缺少 properties 字段的 object 类型 → 补全空 properties
- 名称清洗：`sanitize_mcp_name_component()` 将非法字符（如 `-`）替换为 `_`

**转换链**：`MCP wire format` → `_convert_mcp_schema()` → `registry.register(schema=...)` → `get_definitions()` 输出 OpenAI 格式 → `convert_tools_to_anthropic()` 输出 Anthropic 格式（如果需要）

**专用后台事件循环架构**：

MCP 使用独立的后台 asyncio 事件循环（`_mcp_loop`），运行在 daemon 线程中：

```python
# tools/mcp_tool.py L1170-1233
_mcp_loop: Optional[asyncio.AbstractEventLoop] = None
_mcp_thread: Optional[threading.Thread] = None

def _ensure_mcp_loop():
    """Start the MCP background event loop if not already running."""
    global _mcp_loop, _mcp_thread
    if _mcp_loop is not None and _mcp_loop.is_running():
        return
    _mcp_loop = asyncio.new_event_loop()
    _mcp_loop.set_exception_handler(_mcp_loop_exception_handler)
    _mcp_thread = threading.Thread(target=_mcp_loop.run_forever, daemon=True)
    _mcp_thread.start()
```

所有 MCP 工具调用通过 `_run_on_mcp_loop()` 提交到这个专用 loop，避免与主线程的 asyncio 状态冲突，同时保持 MCP 连接（SSE/stdio）的持久复用。

## 九、错误处理哲学与资源管控

### 9.1 核心原则：工具绝不抛异常

Hermes 工具系统的第一铁律：**所有工具 handler 必须返回 JSON 字符串，绝不向上抛异常**。这是从实际生产经验中总结的设计原则——单个工具的异常不应导致整个 agent 会话崩溃。

**一致的返回格式**（tools/registry.py L456-490）：

```python
# 成功返回
def tool_result(data=None, **kwargs) -> str:
    """Return a JSON result string for tool handlers."""
    if data is not None:
        return json.dumps(data, ensure_ascii=False)
    return json.dumps(kwargs, ensure_ascii=False)

# 错误返回
def tool_error(message, **extra) -> str:
    """Return a JSON error string for tool handlers."""
    result = {"error": str(message)}
    if extra:
        result.update(extra)
    return json.dumps(result, ensure_ascii=False)
```

所有工具文件中使用这两个 helper，消除了数百处重复的 `json.dumps({"error": ...})` 样板代码，同时保证了 LLM 能可靠地解析错误信息。

### 9.2 三层异常安全边界

**第一层 — 工具 handler 内部**：每个 handler 自行捕获预期异常，返回 `tool_error()`。例如：

```python
# 工具内部自己处理可预见的错误
try:
    result = some_operation()
except ValueError as e:
    return tool_error(f"Invalid input: {e}")
return tool_result(result)
```

**第二层 — `registry.dispatch()`**：作为第二道防线，catch 所有未在 handler 内部处理的异常：

```python
def dispatch(self, name, args, **kwargs) -> str:
    try:
        return entry.handler(args, **kwargs)
    except Exception as e:
        return json.dumps({"error": f"Tool execution failed: {e}"})
```

**第三层 — `_invoke_tool()` / `_run_tool()` worker**：run_agent.py 的并发执行路径中有额外的 try/except 包裹，确保 worker 线程不会因异常崩溃：

```python
# run_agent.py — _run_tool() worker
try:
    result = self._invoke_tool(function_name, function_args, ...)
except Exception as tool_error:
    result = f"Error executing tool '{function_name}': {tool_error}"
```

### 9.3 工具失败检测

`_detect_tool_failure()` 在 `agent/display.py` L763 实现，用于在工具执行后判断是否失败：

```python
def _detect_tool_failure(tool_name: str, result: str | None) -> tuple[bool, str]:
    """Inspect a tool result string for signs of failure."""
    # terminal: 检查 exit_code != 0
    # memory: 区分 "full" (超过限制) 和真正的错误
    # 通用: 检查字符串是否包含 "error" / "failed" / 以 "Error" 开头
    return (is_failure, suffix)  # suffix 如 " [exit 1]" 或 " [error]"
```

在并发执行中，`_detect_tool_failure()` 的结果被记录但不阻塞执行——失败的 tool result 仍然作为正常上下文传递给 LLM，让模型自行判断如何处理。

### 9.4 结果大小预算（Result Size Budgeting）

工具结果可能非常大（如 web_extract 返回的完整网页内容），需要预算控制防止上下文窗口溢出。

**`budget_config.py` — 三层预算体系**：

```python
# budget_config.py
PINNED_THRESHOLDS: Dict[str, float] = {
    "read_file": float("inf"),  # read_file 永不截断，防止无限循环
}

DEFAULT_RESULT_SIZE_CHARS: int = 100_000   # 默认单结果上限
DEFAULT_TURN_BUDGET_CHARS: int = 200_000   # 单轮累计预算
DEFAULT_PREVIEW_SIZE_CHARS: int = 1_500    # 持久化后的内联预览

@dataclass(frozen=True)
class BudgetConfig:
    default_result_size: int = DEFAULT_RESULT_SIZE_CHARS
    turn_budget: int = DEFAULT_TURN_BUDGET_CHARS
    preview_size: int = DEFAULT_PREVIEW_SIZE_CHARS
    tool_overrides: Dict[str, int] = field(default_factory=dict)

    def resolve_threshold(self, tool_name: str) -> int | float:
        """优先级：pinned → tool_overrides → registry per-tool → default"""
        if tool_name in PINNED_THRESHOLDS:
            return PINNED_THRESHOLDS[tool_name]
        if tool_name in self.tool_overrides:
            return self.tool_overrides[tool_name]
        return registry.get_max_result_size(tool_name, default=self.default_result_size)
```

**per-tool 粒度控制**：`registry.register()` 支持 `max_result_size_chars` 参数，`get_max_result_size()` 查询：

```python
# tools/registry.py L315
def get_max_result_size(self, name: str, default=None) -> int | float:
    entry = self.get_entry(name)
    if entry and entry.max_result_size_chars is not None:
        return entry.max_result_size_chars
    if default is not None:
        return default
    return DEFAULT_RESULT_SIZE_CHARS  # from budget_config
```

### 9.5 安全辅助模块概览

Hermes 包含多个安全辅助模块，保护 agent 免受恶意提示的攻击：

## 十、面试八股文

### 8.1 什么是注册表模式？优缺点？

**定义**：提供一个全局的、集中的存储库来管理对象的注册和查找。

**优点**：解耦、可扩展、集中管理、运行时灵活性
**缺点**：隐式依赖、注册时序敏感、单点故障、测试复杂

### 8.2 AST 是什么？如何用于代码分析？

AST (Abstract Syntax Tree) 是源代码的树状结构表示。Hermes 用它精确匹配 `registry.register()` 调用，避免正则的假阳性。

### 8.3 插件系统的设计原则

1. 最小接口：只需 `registry.register()` 调用
2. 自描述：schema + handler + check_fn 在同一文件
3. 隔离性：不能覆盖内置工具
4. 可发现性：AST 扫描自动发现
5. 优雅降级：加载失败不影响其他工具

### 8.4 如何实现工具的热插拔？

1. 动态注册/注销
2. 变更通知（MCP `notifications/tools/list_changed`）
3. 原子刷新（asyncio.Lock）
4. 快照隔离（读操作不被写操作阻塞）

### 8.5 单例模式的实现方式和线程安全

| 方式 | 线程安全 | 代码简洁 |
|------|---------|---------|
| 模块级变量 | 天然安全 | 最简洁 |
| `__new__` | 需加锁 | 中等 |
| 装饰器 | 需加锁 | 简洁 |
| 元类 | 需加锁 | 复杂 |

### 8.6 Python 的 importlib 动态导入机制

```python
module = importlib.import_module("tools.terminal_tool")
```

导入的模块被缓存在 `sys.modules` 中，重复导入不会重新执行。

## 十一、面试官可能问的问题及详细回答

### Q1: 请描述 Hermes 工具系统的整体架构

四层架构：工具实现层（自包含 schema + handler）→ 注册中心（单例注册表）→ 工具集层（分组和组合）→ 编排层（发现和桥接）。

### Q2: 为什么选择自注册模式而不是集中配置？

三个关键优势：开闭原则（新增工具无需修改已有代码）、内聚性（schema 和 handler 在同一文件）、自动发现（AST 扫描避免手动维护）。

### Q3: AST 扫描相比正则表达式有什么优势？

AST 理解 Python 语法，不会被注释/字符串干扰，只检查顶层语句避免误匹配，只解析不执行避免副作用。

### Q4: ToolRegistry 如何保证线程安全？

使用 `RLock` 保护修改操作，读操作使用快照隔离模式。MCP 动态刷新不影响正在进行的工具查询。

### Q5: toolset 系统如何处理循环依赖？

使用 `visited` 集合检测循环。兄弟 include 共享同一个 `visited` 集合，确保菱形依赖中每个工具集只解析一次。

### Q6: MCP 工具的动态刷新是如何实现的？

收到 `notifications/tools/list_changed` 通知 → 获取新工具列表 → 注销旧工具 → 注册新工具。使用 `asyncio.Lock` 保证原子性。

### Q7: 子智能体如何避免递归委派？

深度限制（`_delegate_depth >= MAX_DEPTH`）+ 工具层面阻止（`DELEGATE_BLOCKED_TOOLS` 移除 `delegate_task`）。

### Q8: terminal_tool 的策略模式如果要新增一种后端，需要修改哪些代码？

只需两步：(1) 创建新环境文件继承 `BaseEnvironment`；(2) 在 `_create_environment()` 添加 elif 分支。

### Q9: 为什么 Hermes 选择模块级单例？

Python 模块系统天然保证唯一性，无需重写构造逻辑，天然支持 pickle，线程安全由 import lock 保证。

### Q10: file_tools 如何防止 LLM 陷入读取循环？

三层机制：去重检查（mtime 相同返回 stub）、连续读取计数（3 次警告 4 次阻止）、跨工具重置。

### Q11: _run_async 的三种执行路径

1. 已有运行中循环 → 新线程 `asyncio.run()`
2. 工作线程 → 线程本地持久 loop
3. 主线程无循环 → 全局持久 `_tool_loop`

### Q12: 如何评估一个工具框架的可扩展性？

新增工具成本、插件机制、工具集组合、动态性、隔离性、性能。

### Q13: delegate_tool 如何实现凭证继承和轮换？

三层机制：配置覆盖 → 父级继承 → 凭证池共享。支持租约机制自动轮换。

### Q14: MCP 模块如何处理安全问题？

环境变量过滤、凭证脱敏、Prompt injection 检测、OSV 恶意软件检查、碰撞保护。

### Q15: 这个系统有哪些可以改进的地方？

类型安全（Pydantic schema）、依赖声明、性能监控、版本管理、测试支持、文档生成。
