## 一、模块概述

Hermes Agent 的工具系统是一个高度模块化的插件式架构，核心设计哲学：**自注册、自发现**。

### 四层架构
```
┌─────────────────────────────────────────────────────┐
│  model_tools.py  —— 编排层                           │
│  负责触发发现、提供公共API、桥接同步/异步              │
├─────────────────────────────────────────────────────┤
│  toolsets.py  —— 工具集层                            │
│  定义工具分组、递归解析、组合继承                      │
├─────────────────────────────────────────────────────┤
│  tools/registry.py  —— 注册中心                      │
│  单例注册表、schema管理、分发执行                      │
├─────────────────────────────────────────────────────┤
│  tools/*.py  —— 工具实现层                           │
│  每个文件自包含schema+handler+注册调用                 │
└─────────────────────────────────────────────────────┘
```

### 导入链设计（循环导入安全）
```
tools/registry.py  (不导入 model_tools 或工具文件)
       ^
tools/*.py  (在模块级别导入 tools.registry)
       ^
model_tools.py  (导入 tools.registry + 所有工具模块)
       ^
run_agent.py, cli.py, batch_runner.py, etc.
```

### 核心设计优势
1. **开闭原则**：新增工具无需修改现有核心代码
2. **内聚性**：工具的schema、handler、依赖检查在同一文件
3. **自动发现**：AST扫描识别registry.register调用，无需手动维护工具列表
4. **扩展性**：支持MCP外部工具、插件工具动态加载
5. **隔离性**：工具之间不直接依赖，通过注册表通信

## 二、ToolRegistry 详解

### 2.1 单例模式实现

模块级单例：
```python
# registry.py
registry = ToolRegistry()
```

Python 的模块系统天然保证每个模块只被导入一次（`sys.modules` 缓存），因此是天然的单例。线程安全由 import lock 保证。

### 2.2 AST 扫描自动发现机制

```python
def discover_builtin_tools(tools_dir=None):
    tools_path = Path(tools_dir) if tools_dir else Path(__file__).resolve().parent
    module_names = [
        f"tools.{path.stem}"
        for path in sorted(tools_path.glob("*.py"))
        if path.name not in {"__init__.py", "registry.py", "mcp_tool.py"}
        and _module_registers_tools(path)
    ]
```

`_module_registers_tools()` 用 AST 分析精确匹配 `registry.register(...)` 调用模式：

```python
def _is_registry_register_call(node: ast.AST) -> bool:
    if not isinstance(node, ast.Expr) or not isinstance(node.value, ast.Call):
        return False
    func = node.value.func
    return (
        isinstance(func, ast.Attribute)
        and func.attr == "register"
        and isinstance(func.value, ast.Name)
        and func.value.id == "registry"
    )
```

**为什么用 AST 而不是正则**：
1. 精确性：AST 理解 Python 语法，不会被注释、字符串干扰
2. 结构化：只检查顶层语句，避免函数内部调用的误匹配
3. 安全性：只解析不执行，不会触发模块副作用

### 2.3 register() 方法

```python
def register(
    self,
    name: str,           # 工具唯一名称
    toolset: str,        # 所属工具集
    schema: dict,        # OpenAI function-calling 格式的 schema
    handler: Callable,   # 工具执行函数
    check_fn: Callable = None,       # 可用性检查函数
    requires_env: list = None,       # 依赖的环境变量
    is_async: bool = False,          # 是否为异步处理器
    description: str = "",
    emoji: str = "",
    max_result_size_chars: int | float | None = None,
):
```

**防覆盖机制**：当同名工具从不同 toolset 注册时，系统拒绝覆盖内置工具，但允许 MCP 工具之间的覆盖。

```python
# 覆盖规则实现
if existing and existing.toolset != toolset:
    both_mcp = (
        existing.toolset.startswith("mcp-")
        and toolset.startswith("mcp-")
    )
    if both_mcp:
        logger.debug(f"Tool '{name}': MCP toolset '{toolset}' overwriting MCP toolset '{existing.toolset}'")
    else:
        logger.error(f"Tool registration REJECTED: '{name}' (toolset '{toolset}') would shadow existing tool from toolset '{existing.toolset}'")
        return
```

### 2.4 ToolEntry 数据类

使用 `__slots__` 优化内存（每个实例节省约 40% 内存）：

```python
class ToolEntry:
    __slots__ = (
        "name", "toolset", "schema", "handler", "check_fn",
        "requires_env", "is_async", "description", "emoji",

    )
```

### 2.5 Schema 生成和验证

`get_definitions()` 将注册的工具转换为 OpenAI function-calling 格式。使用 `check_results` 缓存机制：同一个 `check_fn` 只调用一次。

```python
def get_definitions(self, tool_names: Set[str], quiet: bool = False) -> List[dict]:
    result = []
    check_results: Dict[Callable, bool] = {}  # check_fn结果缓存
    entries_by_name = {entry.name: entry for entry in self._snapshot_entries()}
    
    for name in sorted(tool_names):
        entry = entries_by_name.get(name)
        if not entry:
            continue
        if entry.check_fn:
            if entry.check_fn not in check_results:
                try:
                    check_results[entry.check_fn] = bool(entry.check_fn())
                except Exception:
                    check_results[entry.check_fn] = False
            if not check_results[entry.check_fn]:
                continue
        schema_with_name = {**entry.schema, "name": entry.name}
        result.append({"type": "function", "function": schema_with_name})
    return result
```

### 2.6 分发执行

```python
def dispatch(self, name: str, args: dict, **kwargs) -> str:
    entry = self.get_entry(name)
    if not entry:
        return json.dumps({"error": f"Unknown tool: {name}"})
    try:
        if entry.is_async:
            from model_tools import _run_async
            return _run_async(entry.handler(args, **kwargs))
        return entry.handler(args, **kwargs)
    except Exception as e:
        return json.dumps({"error": f"Tool execution failed: {e}"})
```

### 2.7 线程安全设计

使用 `threading.RLock()`（可重入锁）保护所有修改操作。读操作使用**快照隔离**模式。

```python
def _snapshot_state(self) -> tuple[List[ToolEntry], Dict[str, Callable]]:
    """返回注册表条目的一致快照，避免读写竞争"""
    with self._lock:
        return list(self._tools.values()), dict(self._toolset_checks)
```

## 三、Toolset System 详解

### 3.1 工具集的组合与继承

两种工具集：

**叶子工具集**（直接工具列表）：
```python
"web": {
    "description": "Web research and content extraction tools",
    "tools": ["web_search", "web_extract"],
    "includes": []
}
```

**组合工具集**（包含其他工具集）：
```python
"debugging": {
    "description": "Debugging and troubleshooting toolkit",
    "tools": ["terminal", "process"],
    "includes": ["web", "file"]  # 递归包含
}
```

### 3.2 递归解析机制

```python
def resolve_toolset(name: str, visited: Set[str] = None) -> List[str]:
    if visited is None:
        visited = set()
    if name in {"all", "*"}:
        all_tools = set()
        for toolset_name in get_toolset_names():
            resolved = resolve_toolset(toolset_name, visited.copy())
            all_tools.update(resolved)
        return sorted(all_tools)
    if name in visited:
        return []  # 循环检测
    visited.add(name)
    toolset = get_toolset(name)
    tools = set(toolset.get("tools", []))
    for included_name in toolset.get("includes", []):
        included_tools = resolve_toolset(included_name, visited)
        tools.update(included_tools)
    return sorted(tools)
```

关键设计：循环检测、菱形依赖安全、"all" 别名。

### 3.3 工具集动态扩展

工具集不仅支持静态定义，还支持插件和MCP动态添加：

```python
def get_all_toolsets() -> Dict[str, Dict[str, Any]]:
    result = dict(TOOLSETS)
    aliases = _get_registry_toolset_aliases()
    for ts_name in _get_plugin_toolset_names():
        # 动态添加插件工具集

        for alias, canonical in aliases.items():
            if canonical == ts_name and alias not in TOOLSETS:
                display_name = alias
                break
        if display_name in result:

        toolset = get_toolset(display_name)
        if toolset:
            result[display_name] = toolset
    return result
```

## 四、编排层 (model_tools.py) 核心实现

### 4.1 异步桥接设计

解决同步上下文调用异步工具的问题，支持三种执行路径：

```python
def _run_async(coro):
    """Run an async coroutine from a sync context."""
    try:
        loop = asyncio.get_running_loop()
    except RuntimeError:
        loop = None

    if loop and loop.is_running():
        # 1. 已有运行中循环 → 新线程异步执行
        import concurrent.futures
        with concurrent.futures.ThreadPoolExecutor(max_workers=1) as pool:
            future = pool.submit(asyncio.run, coro)
            return future.result(timeout=300)

    if threading.current_thread() is not threading.main_thread():
        # 2. 工作线程 → 线程本地持久loop
        worker_loop = _get_worker_loop()
        return worker_loop.run_until_complete(coro)

    # 3. 主线程无循环 → 全局持久tool_loop
    tool_loop = _get_tool_loop()
    return tool_loop.run_until_complete(coro)
```

**持久Loop设计优势**：避免重复创建关闭loop导致的"Event loop is closed"错误，保持HTTP客户端连接复用。

### 4.3 工具参数类型强制

LLM经常返回字符串形式的数字/布尔值，coerce_tool_args自动转换为schema声明的类型：

```python
def coerce_tool_args(tool_name: str, args: Dict[str, Any]) -> Dict[str, Any]:
    """Coerce tool call arguments to match their JSON Schema types."""
    if not args or not isinstance(args, dict):
        return args

    schema = registry.get_schema(tool_name)
    if not schema:
        return args

    properties = (schema.get("parameters") or {}).get("properties")
    if not properties:
        return args

    for key, value in args.items():
        if not isinstance(value, str):
            continue
        prop_schema = properties.get(key)
        if not prop_schema:
            continue
        expected = prop_schema.get("type")
        if not expected:
            continue
        coerced = _coerce_value(value, expected)
        if coerced is not value:
            args[key] = coerced

    return args
```

支持类型：
- integer: "42" → 42
- number: "3.14" → 3.14
- boolean: "true" → True, "false" → False
- 联合类型：尝试所有类型直到成功

## 五、工具实现层规范

### 5.1 最简工具示例 (clarify_tool.py)

```python
#!/usr/bin/env python3
"""Clarify Tool Module - Interactive Clarifying Questions"""

import json
from typing import List, Optional, Callable
from tools.registry import registry, tool_error

MAX_CHOICES = 4

def clarify_tool(
    question: str,
    choices: Optional[List[str]] = None,
    callback: Optional[Callable] = None,
) -> str:
    """Ask the user a question, optionally with multiple-choice options."""
    if not question or not question.strip():
        return tool_error("Question text is required.")
    
    question = question.strip()
    
    # Validate choices
    if choices is not None:
        choices = [str(c).strip() for c in choices if str(c).strip()]
        if len(choices) > MAX_CHOICES:
            choices = choices[:MAX_CHOICES]
        if not choices:
            choices = None
    
    if callback is None:
        return json.dumps(
            {"error": "Clarify tool is not available in this execution context."},
            ensure_ascii=False,
        )
    
    try:
        user_response = callback(question, choices)
    except Exception as exc:
        return json.dumps(
            {"error": f"Failed to get user input: {exc}"},
            ensure_ascii=False,
        )
    
    return json.dumps({
        "question": question,
        "choices_offered": choices,
        "user_response": str(user_response).strip(),
    }, ensure_ascii=False)

def check_clarify_requirements() -> bool:
    """Clarify tool has no external requirements -- always available."""
    return True

# OpenAI Function-Calling Schema
CLARIFY_SCHEMA = {
    "name": "clarify",
    "description": "Ask the user a question when you need clarification...",
    "parameters": {
        "type": "object",
        "properties": {
            "question": {
                "type": "string",
                "description": "The question to present to the user.",
            },
            "choices": {
                "type": "array",
                "items": {"type": "string"},
                "maxItems": MAX_CHOICES,
                "description": "Up to 4 answer choices...",
            },
        },
        "required": ["question"],
    },
}

# 注册工具
registry.register(
    name="clarify",
    toolset="clarify",
    schema=CLARIFY_SCHEMA,
    handler=lambda args, **kw: clarify_tool(
        question=args.get("question", ""),
        choices=args.get("choices"),
        callback=kw.get("callback")),
    check_fn=check_clarify_requirements,
    emoji="❓",
)
```

### 5.2 工具实现规范

每个工具文件必须包含：
1. **Schema定义**：遵循OpenAI function-calling格式
2. **Handler函数**：工具实际执行逻辑，返回JSON字符串
3. **Check函数**：可用性检查，返回bool
4. **注册调用**：调用registry.register完成自注册

### 5.3 工具返回格式规范

所有工具必须返回JSON字符串，使用registry提供的辅助函数：

```python
# 成功返回
return tool_result({"key": "value"})
# 或
return tool_result(success=True, data=payload)

# 错误返回
return tool_error("Something went wrong")
# 或带额外信息
return tool_error("Permission denied", code=403)
```

## 七、设计模式分析

| 模式 | 应用位置 | 作用 |
|------|---------|------|
| 注册表模式 | `ToolRegistry` | 集中管理工具注册和发现，解耦工具提供者和消费者 |
| 策略模式 | 终端工具的 6 种后端 | 运行时切换不同执行环境，无需修改核心逻辑 |
| 工厂模式 | `_create_environment()`, `_make_tool_handler()` | 封装对象创建逻辑，客户端无需关心具体实现 |
| 观察者模式 | MCP 的 `notifications/tools/list_changed` | 工具列表变化时自动通知相关组件更新 |
| 模板方法 | `BaseEnvironment` 定义骨架 | 定义环境生命周期模板，子类实现具体步骤 |
| 组合模式 | toolset 系统的递归解析 | 单个工具和工具集使用统一接口，支持任意深度组合 |
| 享元模式 | `check_results` 缓存 | 复用同一check_fn的执行结果，避免重复调用 |
| 快照模式 | registry读操作隔离 | 读操作不受并发写影响，保证数据一致性 |

## 八、面试八股文

### 8.1 什么是注册表模式？优缺点？

**定义**：提供一个全局的、集中的存储库来管理对象的注册和查找。

**优点**：解耦、可扩展、集中管理、运行时灵活性
**缺点**：隐式依赖、注册时序敏感、单点故障、测试复杂

### 8.2 AST 是什么？如何用于代码分析？

AST (Abstract Syntax Tree) 是源代码的树状结构表示。Hermes 用它精确匹配 `registry.register()` 调用，避免正则的假阳性。

### 8.3 插件系统的设计原则

1. 最小接口：只需 `registry.register()` 调用
2. 自描述：schema + handler + check_fn 在同一文件
3. 隔离性：不能覆盖内置工具
4. 可发现性：AST 扫描自动发现
5. 优雅降级：加载失败不影响其他工具

### 8.4 如何实现工具的热插拔？

1. 动态注册/注销
2. 变更通知（MCP `notifications/tools/list_changed`）
3. 原子刷新（asyncio.Lock）
4. 快照隔离（读操作不被写操作阻塞）

### 8.5 单例模式的实现方式和线程安全

| 方式 | 线程安全 | 代码简洁 |
|------|---------|---------|
| 模块级变量 | 天然安全 | 最简洁 |
| `__new__` | 需加锁 | 中等 |
| 装饰器 | 需加锁 | 简洁 |
| 元类 | 需加锁 | 复杂 |

### 8.6 Python 的 importlib 动态导入机制

```python
module = importlib.import_module("tools.terminal_tool")
```

导入的模块被缓存在 `sys.modules` 中，重复导入不会重新执行。

## 九、面试官可能问的问题及详细回答

### Q1: 请描述 Hermes 工具系统的整体架构

四层架构：工具实现层（自包含 schema + handler）→ 注册中心（单例注册表）→ 工具集层（分组和组合）→ 编排层（发现和桥接）。

### Q2: 为什么选择自注册模式而不是集中配置？

三个关键优势：开闭原则（新增工具无需修改已有代码）、内聚性（schema 和 handler 在同一文件）、自动发现（AST 扫描避免手动维护）。

### Q3: AST 扫描相比正则表达式有什么优势？

AST 理解 Python 语法，不会被注释/字符串干扰，只检查顶层语句避免误匹配，只解析不执行避免副作用。

### Q4: ToolRegistry 如何保证线程安全？

使用 `RLock` 保护修改操作，读操作使用快照隔离模式。MCP 动态刷新不影响正在进行的工具查询。

### Q5: toolset 系统如何处理循环依赖？

使用 `visited` 集合检测循环。兄弟 include 共享同一个 `visited` 集合，确保菱形依赖中每个工具集只解析一次。

### Q6: MCP 工具的动态刷新是如何实现的？

收到 `notifications/tools/list_changed` 通知 → 获取新工具列表 → 注销旧工具 → 注册新工具。使用 `asyncio.Lock` 保证原子性。

### Q7: 子智能体如何避免递归委派？

深度限制（`_delegate_depth >= MAX_DEPTH`）+ 工具层面阻止（`DELEGATE_BLOCKED_TOOLS` 移除 `delegate_task`）。

### Q8: terminal_tool 的策略模式如果要新增一种后端，需要修改哪些代码？

只需两步：(1) 创建新环境文件继承 `BaseEnvironment`；(2) 在 `_create_environment()` 添加 elif 分支。

### Q9: 为什么 Hermes 选择模块级单例？

Python 模块系统天然保证唯一性，无需重写构造逻辑，天然支持 pickle，线程安全由 import lock 保证。

### Q10: file_tools 如何防止 LLM 陷入读取循环？

三层机制：去重检查（mtime 相同返回 stub）、连续读取计数（3 次警告 4 次阻止）、跨工具重置。

### Q11: _run_async 的三种执行路径

1. 已有运行中循环 → 新线程 `asyncio.run()`
2. 工作线程 → 线程本地持久 loop
3. 主线程无循环 → 全局持久 `_tool_loop`

### Q12: 如何评估一个工具框架的可扩展性？

新增工具成本、插件机制、工具集组合、动态性、隔离性、性能。

### Q13: delegate_tool 如何实现凭证继承和轮换？

三层机制：配置覆盖 → 父级继承 → 凭证池共享。支持租约机制自动轮换。

### Q14: MCP 模块如何处理安全问题？

环境变量过滤、凭证脱敏、Prompt injection 检测、OSV 恶意软件检查、碰撞保护。

### Q15: 这个系统有哪些可以改进的地方？

类型安全（Pydantic schema）、依赖声明、性能监控、版本管理、测试支持、文档生成。
